# System Architecture & Threat Model

## 1. 신뢰 경계 (Trust Boundary)
본 시스템은 다음과 같이 신뢰 영역을 격격히 분리합니다.

- **[Client Side]**: 입찰자의 브라우저 또는 CLI. 입찰가의 평문을 알고 있으며, Double-Gate(암호화 + 커미트먼트)를 생성하는 영역.
- **[API Gateway / Server]**: 입찰 데이터를 수신하고 서버 서명 영수증을 발급. 입찰가 평문을 알 수 없으며, 데이터를 삭제하거나 수정할 수 없는 Append-only 저장소 운영.
- **[KMS / HSM Domain]**: 마스터 키(Root KEK)가 거주하며, 시간 정책 및 다중 관리자 승인을 집행하는 최상위 신뢰 영역.
- **[SQLite Storage]**: 암호화된 데이터 및 이력(Version)이 저장되는 비신뢰 영역.

## 2. 위협 모델 (Threat Model)
| 위협 유형 | 시나리오 | 대응책 |
| :--- | :--- | :--- |
| **내부자 위협** | DBA/관리자가 DB의 모든 암호문을 탈취 | 경매 개인키가 KMS KEK로 암호화되어 있으며, 다중 관리자 승인 없이 복제 불가. |
| **조기 복호화** | 마감 전 서버 취약점을 통해 복호화 시도 | KMS 정책에 "현재 시간 > 마감 시간" 및 "N-of-M 승인" 조건을 결합하여 키 방출 거부. |
| **데이터 변조** | 공격자가 DB의 특정 입찰가를 변조/삭제 | AES-GCM Auth Tag, 사용자 서명 및 Append-only 구조(상태 전이 이력)로 감지. |
| **사용자 부인** | 입찰 후 "내가 입찰하지 않았다"고 주장 | 입찰 시 사용자의 개인키로 서명된 페이로드와 서버 서명 영수증(Receipt)으로 상호 확약. |
| **중간자 공격 (MITM)** | 가짜 서버가 사용자의 공개키를 가로챔 | 공개키 지문(Fingerprint) 시스템을 통해 사용자가 등록된 키의 무결성을 직접 확인. |

## 3. Double-Gate 입찰 메커니즘
단순 암호화를 넘어 입찰의 무결성을 이중으로 보호합니다.
1.  **Gate 1 (기밀성)**: RSA-OAEP로 입찰가와 대칭키를 암호화하여 서버가 내용을 볼 수 없게 함.
2.  **Gate 2 (무결성)**: 입찰가와 Nonce를 조합한 SHA-256 해시(Commitment)를 생성하여, 나중에 공개(Reveal) 시 데이터가 변조되지 않았음을 증명.

## 4. Append-only 이력 관리 (Traceability)
모든 입찰은 삭제되거나 수정되지 않습니다.
- 동일 사용자가 재입찰 시, 기존 입찰은 `DEPRECATED` 상태로 전이되며 새로운 `v2`, `v3` 입찰이 생성됩니다.
- 모든 버전은 `parent_bid_id`를 통해 체인처럼 연결되어 입찰의 전체 수정 이력을 추적할 수 있습니다.

## 5. KMS 다중 승인 및 시간 정책 (N-of-M)
경매 마감 후 키를 해제하기 위해 다음 두 조건을 동시에 충족해야 합니다.
1.  **시간 조건**: 현재 시간 > 경매 마감 시간.
2.  **승인 조건**: 최소 $N$명 이상의 관리자가 디지털 서명한 승인 토큰(Approval Token) 제출.
